## 🔧 PROBLEMA DETECTADO Y SOLUCIONADO

### Problema Original
La imagen `hellohal2064/vllm-dgx-spark-gb10:latest` tenía un entrypoint hardcodeado para un modelo diferente (`Qwen3-Next-80B-A3B-Thinking-FP8`) que sobrescribía los argumentos del docker-compose.yml.

### Solución Aplicada
Cambiado a imagen oficial: **`vllm/vllm-openai:latest`**

Esta imagen:
- Es mantenida oficialmente por el proyecto vLLM
- No tiene entrypoint custom que sobrescriba argumentos
- Funciona con cualquier modelo compatible
- Es más estable y actualizada

### docker-compose.yml Final (CORRECTO)

```yaml
services:
  vllm-coder:
    image: vllm/vllm-openai:latest  # ⬅ CAMBIADO
    container_name: vllm-coder
    restart: unless-stopped
    runtime: nvidia
    shm_size: 4g
    ports:
      - "8001:8000"
    volumes:
      - /home/daviddgx/models:/models:ro
      - vllm-cache:/root/.cache
    environment:
      - HF_HOME=/root/.cache/huggingface
      - VLLM_API_KEY=${VLLM_API_KEY}
    command:
      - --model
      - /models/qwen2.5-coder-32b-awq
      - --served-model-name
      - qwen2.5-coder-32b
      - --quantization
      - awq  # ⬅ CORREGIDO (no awq_marlin)
      - --dtype
      - half
      - --max-model-len
      - "32768"
      - --gpu-memory-utilization
      - "0.85"
      - --max-num-seqs
      - "8"
      - --disable-log-requests
    healthcheck:
      test: ["CMD", "curl", "-fs", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 180s

volumes:
  vllm-cache:
    driver: local

networks:
  default:
    name: seedy_default
    external: true
```

## 🔧 TROUBLESHOOTING

### Problema: Contenedor crashea al arrancar

**Logs:**
```bash
docker compose logs vllm-coder
```

**Errores comunes:**

#### 1. "CUDA out of memory"
```
RuntimeError: CUDA out of memory.
```

**Solución:**
```bash
# Editar docker-compose.yml
nano ~/seedy/coder-vllm/docker-compose.yml

# Cambiar:
--gpu-memory-utilization 0.85
# Por:
--gpu-memory-utilization 0.70

# Reiniciar
docker compose down && docker compose up -d
```

#### 2. "Model not found"
```
ValueError: Model not found at /models/qwen2.5-coder-32b-awq
```

**Solución:**
```bash
# Verificar mount
docker exec vllm-coder ls -la /models/qwen2.5-coder-32b-awq

# Si vacío, verificar path en host
ls -la ~/models/qwen2.5-coder-32b-awq

# Corregir path en docker-compose.yml si necesario
```

#### 3. "Port 8001 already in use"
```
Error: bind: address already in use
```

**Solución:**
```bash
# Ver qué usa el puerto
sudo lsof -i :8001

# Matar proceso
sudo kill -9 <PID>

# O cambiar puerto en docker-compose.yml
ports:
  - "8002:8000"  # Cambiar 8001 → 8002
```

### Problema: vLLM muy lento (>30s)

**Diagnóstico:**
```bash
# Monitorear GPU mientras hace inference
nvidia-smi dmon -s u -c 10
```

**Si GPU util <50%:**
- Problema: memory bandwidth ARM64 limitado
- Solución: aceptar latency o usar fallback Together.ai

**Si GPU util >90%:**
- Problema: modelo muy grande para memoria disponible
- Solución: bajar `max_model_len` de 32768 a 16384

### Problema: Ollama se ralentiza con vLLM

**Síntomas:**
- Ollama timeout cuando vLLM activo
- Errores OOM en logs Ollama

**Solución 1: Bajar vLLM memory**
```bash
# docker-compose.yml
--gpu-memory-utilization 0.60  # Era 0.85

docker compose down && docker compose up -d
```

**Solución 2: Usar vLLM on-demand**
```bash
# Apagar vLLM cuando no se use
docker compose stop

# Encender solo cuando se necesite
docker compose start
```

**Solución 3: Reiniciar Ollama**
```bash
docker restart ollama
```

## 🎓 LECCIONES APRENDIDAS

### 1. Imágenes Docker Custom vs Oficiales

**Problema:** Imagen custom `hellohal2064/vllm-dgx-spark-gb10` tenía entrypoint hardcodeado que sobrescribía argumentos.

**Lección:** Siempre preferir imágenes oficiales mantenidas por el proyecto. Si se necesita custom, revisar entrypoint/CMD antes de usar.

### 2. ARM64 GB10 Limitations

**Problema:** Memory bandwidth limitado en GB10 ARM64 → 4.3 tok/s con qwen2.5:72b local.

**Lección:** Modelos 32B+ en GB10 son viables pero lentos (15-22 tok/s). Para tareas interactivas, preferir 7-14B. Para análisis batch async, 32B+ OK.

### 3. Docker Networking en Multi-Host

**Problema:** Seedy backend (MSI) necesita acceder a vLLM (DGX) pero ambos en redes Docker diferentes.

**Solución:** vLLM publica puerto 8001 en host DGX → MSI accede via IP DGX (192.168.20.57:8001).

**Lección:** Para servicios multi-host, publicar puertos en host y usar IPs host, no nombres de contenedor.

### 4. Quantization AWQ vs GGUF

**Observación:** AWQ (AutoAWQ Int4) mejor throughput en GB10 que GGUF Q4_K_M equivalente.

**Razón:** AWQ optimizado para GPUs (CUDA kernels), GGUF optimizado para CPU.

**Lección:** Para deployments GPU, preferir AWQ/GPTQ. Para CPU, GGUF.
