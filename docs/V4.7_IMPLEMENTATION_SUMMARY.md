### 2.1 VLLMLocalProvider (NUEVO)

**Archivo:** `backend/coder/providers/vllm_local_provider.py`

```python
class VLLMLocalProvider(CoderProvider):
    """
    Provider para vLLM local en puerto 8001 sirviendo Qwen2.5-Coder-32B-AWQ.
    Compatible con OpenAI API format (v1/chat/completions).
    """
    name = "vllm_local"
    
    MODEL_MAP = {
        "vllm:qwen2.5-coder-32b": "qwen2.5-coder-32b",
        "vllm:coder-32b": "qwen2.5-coder-32b",
        "vllm:default": "qwen2.5-coder-32b",
    }
    
    # Funciones:
    - stream(): Chat completions (OpenAI format)
    - _stream_fim(): Fill-in-middle con formato Qwen
    - health_check(): Verificar /v1/models
    - estimate_cost(): $0 (local)
```

**Features:**
- OpenAI-compatible client (httpx async)
- Streaming con SSE parsing
- Timeout 180s (32B es lento, ~15-22 tok/s esperado)
- FIM support para autocompletion

## 9. TROUBLESHOOTING

### Problema: vLLM OOM

**Síntoma:** `RuntimeError: CUDA out of memory`

**Solución:**
1. Bajar `--gpu-memory-utilization` de 0.85 a 0.75
2. Reducir `--max-num-seqs` de 8 a 4
3. Verificar Ollama no tiene modelos grandes cargados (`docker exec ollama ollama ps`)

### Problema: vLLM lento (>30s/inference)

**Síntoma:** Timeout en health_check, respuestas lentas

**Diagnóstico:**
- GB10 ARM64 tiene memory bandwidth limitado (no es la GPU)
- Qwen2.5-Coder-32B-AWQ es grande para ARM64

**Solución:**
1. Verificar AWQ quantization activa (`--dtype auto`)
2. Considerar FP16 en lugar de AWQ si disponible
3. Fallback a Together.ai (degradation chain ya configurada)

### Problema: Ollama afectado por vLLM

**Síntoma:** Ollama responde lento o falla cuando vLLM activo

**Solución:**
1. Reiniciar Ollama: `docker restart ollama`
2. Bajar `gpu_memory_utilization` de vLLM a 0.70
3. Usar vLLM solo on-demand (no 24/7 como Ollama)
