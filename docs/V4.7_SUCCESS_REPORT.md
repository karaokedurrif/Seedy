## 🔧 PROBLEMAS RESUELTOS DURANTE IMPLEMENTACIÓN

### Problema 1: Docker custom image con entrypoint hardcodeado

**Síntoma:**
```
vllm: error: unrecognized arguments: /models/Qwen3-Next-80B-A3B-Thinking-FP8
```

**Causa:** Imagen `hellohal2064/vllm-dgx-spark-gb10:latest` tenía entrypoint hardcodeado que ignoraba `command:` del docker-compose.

**Solución:** Migrar a imagen oficial `vllm/vllm-openai:latest` sin entrypoint custom.

**Commit:** b83c7d0

---

### Problema 2: Argumento inválido `--disable-log-requests`

**Síntoma:**
```
vllm: error: unrecognized arguments: --disable-log-requests
```

**Causa:** Argumento no existe en vLLM v0.20.1 oficial.

**Solución:** Eliminar línea del docker-compose.yml.

**Commit:** (inline fix)

---

### Problema 3: Out of Memory con gpu-memory-utilization 0.85

**Síntoma:**
```
ValueError: Free memory on device cuda:0 (85.2/121.63 GiB) is less than 
desired GPU memory utilization (0.85, 103.38 GiB)
```

**Causa:** Ollama ya ocupaba 31 GB con 3 modelos cargados:
- qwen2.5:7b (10.1 GB)
- seedy:v16 (20.5 GB)
- mxbai-embed-large (0.7 GB)

**Cálculo:**
- Memory libre: 85.2 GB
- vLLM intenta reservar: 0.85 × 121.63 = 103.38 GB
- 85.2 < 103.38 → ERROR ❌

**Solución:** Bajar `--gpu-memory-utilization` a **0.60**:
- vLLM reserva: 0.60 × 121.63 = 73 GB
- 85.2 > 73 → SUCCESS ✅

**Commit:** (inline fix)

## 🎓 LECCIONES APRENDIDAS

### 1. Docker Images

**Preferir imágenes oficiales** sobre custom builds. Las imágenes custom pueden tener entrypoints hardcodeados que ignoran `command:` en docker-compose.

**Evidencia:** `hellohal2064/vllm-dgx-spark-gb10:latest` tenía entrypoint para modelo diferente, causando fallo silencioso de args.

### 2. GPU Memory Planning

**En arquitectura compartida (Ollama + vLLM), calcular footprint real** antes de configurar `--gpu-memory-utilization`.

**Fórmula correcta:**
```
VLLM_GPU_UTIL = (GPU_TOTAL - OLLAMA_FOOTPRINT) / GPU_TOTAL
```

**Ejemplo GB10:**
- GPU total: 128 GB
- Ollama footprint: 31 GB (3 modelos activos)
- Disponible para vLLM: 97 GB
- GPU util seguro: 0.60 (→ 77 GB reserved, margen 20 GB)

### 3. ARM64 Memory Bandwidth

**Unified memory ARM64** (GB10) muestra `nvidia-smi` con valores "N/A" — esto es **normal y esperado**. No indica error.

**Verificación correcta:** `curl -s localhost:11434/api/ps` (Ollama) para ver modelos cargados y tamaños.

### 4. Primera Inference Lenta

**Primera inference tarda 10-20× más** (torch.compile + CUDA graph compilation). **No es un bug.**

**Solución:** Warmup automático al startup (vLLM ya lo hace), o ejecutar test simple post-deployment.
